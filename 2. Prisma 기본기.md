# 01. 프로젝트 셋업과 Prisma 초기화
<details>
  <summary>(1) Prisma란?</summary>

	```
	Prisma = 차세대 Node.js ORM

	특징:
	- 타입 안전성 (TypeScript 지원)
	- 자동 완성
	- 직관적인 API
	- 마이그레이션 관리
	- 여러 데이터베이스 지원 (PostgreSQL, MySQL, SQLite, MongoDB 등)

	Mongoose vs Prisma:
	Mongoose: MongoDB 전용, 스키마 기반
	Prisma: 여러 DB 지원, 타입 안전, SQL/NoSQL 모두
	```
</details>

# 03. 마이그레이션
* 이전까지 dp push로 schema.prisma를 DB에 반영해왔음
* But, 이 방법은 **변경 이력이 남지 않음**
* 이러한 한계를 보완할 수 방법이 마이그레이션(**`migrate dev`**)
<details>
	<summary>(1) 마이그레이션이란?</summary>

```
마이그레이션 = 스키마 변경 사항을 데이터베이스에 적용

과정:
1. schema.prisma 수정
2. 마이그레이션 생성
3. 데이터베이스에 적용

장점:
- 버전 관리
- 롤백 가능
- 팀 협업 용이
```
</details>

<details>
	<summary>(2) 첫 번째 마이그레이션</summary>

**schema.prisma**
```prisma
model User {
	id				Int			@id @default(autoincrement())
	name			String
	createdAt		DateTime	@default(now())
	updatedAt		DateTime	@updatedAt

	@@map("users")
}
```

**마이그레이션 생성**
```
npx prisma migrate dev --name init
```

**생성되는 것들**
```
prisma/
├── schema.prisma
└── migrations/
    └── 20240101000000_init/
        └── migration.sql
```

**migration.sql**
```SQL
-- CreateTable
CREATE TABLE "users" (
		"id"	SERIAL	NOT NULL,
		"name"	TEXT	NOT NULL,
		"createdAt"	TIMESTAMP(3)	NOT NULL DEFAULT CURRENT_TIMESTAMP,
		"updatedAt"	TIMESTAMP(3)	NOT NULL,

		CONSTRAINT	"users_pkey"	PRIMARY KEY ("id")
);
```
</details>

<details>
	<summary>(3) 스키마 수정 후 마이그레이션</summary>

<details>
	<summary>(4) 마이그레이션 명령어 정리</summary>

|명령어|용도|환경|마이그레이션 파일|
|---|---|---|---|
|`prisma db push`|스키마를 DB에 즉시 반영|로컬(프로토타이핑)|X(안 만듦)|
|`prisma migrate dev`|마이그레이션 파일 생성+적용|로컬 개발|O(만듦)|
|`prisma migrate deploy`|만들어진 마이그레이션을 적용만|운영/스테이징|X(안 만듦, 적용만)|
|`prisma migrate reset`|DB 통째로 리셋 후 재적용|**로컬 전용**|(모두 재적용)|
|`prisma migrate status`|마이그레이션 상태 확인|모든 환경|-|
</details>

	<summary>2. Read - 데이터 조회하기</summary>

#### (1) 이론: findMany / findUnique / findFirst
```JS
// 모든 데이터 조회
const todos = await prisma.todo.findMany();

// 고유 식별자(PK, UNIQUE)로 1개 조회
const todo = await prisma.todo.findUnique({
	where: { id: 1 }
});

// 조건에 맞는 첫 번째 1개 조회(정렬 가능)
const todo = await prisma.todo.findFirst({
	where: { isDone: false },
	orderBy: { createdAt: 'desc' }
});
```
**언제 어느 걸 쓰나?**
* **`findMany`**: 목록 조회(배열 반환, 없으면 빈 배열)
* **`findUnique`**: id처럼 unique한 값으로 정확히 1개(없으면 null)
* **`findFirst`**: 조건에 맞는 첫 번째 1개(where + orderBy 조합)

// 단일 수정
export const updateTodo = async (req, res) => {
	try {
		const { id } = req.params;
		const todo = await prisma.todo.update({
			where: { id: parseInt(id) },
			data: req.body
		});
		res.json({ success: true, data: todo });
	} catch (error) {
		if (error.code === 'P2025') {
			return res.status(404).json({ success: false, messsage: 'Todo를 찾을 수 없습니다' });
		}
		res.status(400).json({ success: false, message: error.message });
	}
};

// 모든 미완료 Todo 일괄 완료 처리
export const completeAllTodos = async (req, res) => {
	try {
		const result = await prisma.todo.updateMany({
			where: { isDone: false },
			data: { isDone: true }
		});
		res.json({ success: true, count: result.count });
	} catch (error) {
		res.status(400).json({ success: false, message: error.message });
	}
};

// Upsert
export const upsertTodo = async (req, res) => {
	try {
		const { id, title, content, isDone } = req.body;
		const todo = await prisma.todo.upsert({
			where: { id },
			update: { title, content, isDone },
			create: { title, content, isDone: isDone || false }
		});
		res.json({ success: true, data: todo });
	} catch (error) {
		res.status(400).json({ success: false, message: error.message });
	}
};

**무엇이 문제?**
* 새 컨트롤러 추가할 때마다 try-catch 복붙
* Prisma 에러코드 분기 처리도 똑같이 복붙
* 한 번이라도 깜빡하면 처리 안 된 에러로 서버가 죽을 수 있음
* 에러 응답 포맷이 함수마다 다를 위험

**해결 방향**: try-catch와 에러 종류별 응답 처리를 **`asyncHandler`라는 함수 한 곳에 모으기**
</details>

	<summary>2. asyncHandler 만들기</summary>

#### (1) src/utils/asyncHandler.js
```JS
import { Prisma } from '@prisma/client';
import { HttpError} from './error.js';
// asyncHandler()의 실행 결과는??
// const asyncHandler = (fn) => async () => {}
app.get("/todos", asyncHandler(createTodo));
export const asynchHandler = (handler) => async (req, res, next) => {
	try {
		// 진짜 컨트롤러 실행
		await handler(req, res, next);
	} catch (error) {
		// 1) 우리가 직접 던진 HTTP 에러 (NotFoundError 등)
		if (error instanceof HttpError) {
			return res.status(error.statusCode).json({
				success: false,
				message: error.message
			});
		}

		// 2) Prisma가 던진 알려진 에러(코드별 매핑)
		if (error instanceof Prisma.PrismaClientKnownRequestError) {
			// P2025: 찾을 행이 없음
			if (error.code === 'P2025') {
				return res.status(404).json({
					success: false,
					message: '리소스를 찾을 수 없습니다'
				});
			}

			// P2002: UNIQUE 제약 위반
			if (error.code === 'P2002') {
				return res.status(409).json({
					success: false,
					message: '이미 존재하는 데이터입니다',
					field: error.meta?.target
				});
			}

			// P2003: 외래 키 제약 위반
			if (erroe.code === 'P2003') {
				return res.status(400).json({
					success: false,
					message: '참조 무결성 제약 조건 위반'
				});
			}
		}

		// 3) Prisma validation 에러(타입 불일치 등)
		if (error instanceof Prisma.PrismaClientValidationError) {
			return res.status(400).json({
				success: false,
				message: 'Prisma validation 에러',
				detail: error.message.split('\n').slice(-2).join(' ')
			});
		}

		// 4) 그 외 알 수 없는 에러 -> 서버 로그에 전체 남기고, 사용자에겐 500
		console.error(error);
		res.status(500.json({
			success: false,
			message: '서버 에러가 발생했습니다'
		});
	}
};
```

#### (2) 어떻게 동작하나? - 한 줄씩 따라가보기
```
1. 컨트롤러를 asyncHandler 로 감싸면
   asyncHandler(async (req, res) => { ... })
   ↓
   asyncHandler 가 새 함수를 반환 (이게 Express 에 등록됨)

2. 요청이 들어오면 새 함수가 실행되면서
   - try 블록 안에서 진짜 핸들러를 await
   - 핸들러 안에서 에러가 던져지면 catch 가 잡음

3. catch 에서 에러 종류 확인
   - HttpError 인스턴스인가? → 그 상태코드로 응답
   - Prisma 알려진 에러인가? → 코드별 매핑된 응답
   - 그 외? → 500

→ 컨트롤러는 "성공 흐름"만 작성, 에러는 throw 만 하면 끝.
```
