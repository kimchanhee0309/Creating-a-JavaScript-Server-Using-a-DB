# 1. User-Todo 일대다(1:N) 관계 정의하기
<details>
  <summary>1. 스키마 수정</summary>

#### (1) 관계 필드 추가
**prisma/schema.prisma**
```prisma
model User {
  id         Int       @id  @default(autoincrement())
  name       String
  todos      Todo[] // <- 새로 추가: "이 user는 여러 Todo를 가진다"
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt

  @@map("users")
}

model Todo {
  id         Int       @id  @default(autoincrement())
  title      String
  content    String?
  isDone     Boolean   @default(false)
  userId     Int       // <- 새로 추가: 외래 키
  user       User      @relation(fields: [userId], references: [id]) // <- 새로 추가: 관계 정의
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt

  @@map("todos")
```

#### (2) 새로 추가된 라인 해설
```
User 내 :
  todos  Todo[]
  → "이 User는 여러(Todo[]) Todo를 소유한다"
  → 1쪽 면. 실제 DB 테이블에는 이 컬럼이 안 생김 (가상 필드)
  → 코드에서 prisma.user.findUnique({ include: { todos: true } }) 쓸 때 필요

Todo 내 :
  userId  Int
  → 외래 키. 실제 DB todos 테이블에 INTEGER 컬럼으로 생김
  
  user    User @relation(fields: [userId], references: [id])
  → "userId 를 users.id 와 연결해라"
  → N쪽 면. 이 관계 설정을 통해 SQL FOREIGN KEY 가 생성됨
  → 코드에서 todo.user 로 소유자 조회 가능
```

#### (3) 마이그레이션
```
npx prisma migrate dev --name add_user_todo_relation
```

**생성되는 SQL(migration.sql)**
```SQL
-- AlterTable
ALTER TABLE "todos" ADD COLUMN "userId" INTEGER NOT NULL;

-- AddForeignKey
ALTER TABLE "todos"
  ADD CONSTRAINT "todos_userId_fkey"
  FOREIGN KEY ("userId") REFERENCES "users"("id")
  ON DELETE RESTRICT ON UPDATE CASCADE;
```
</details>

# 2. Todo-Tag 다대다(N:M) 관계 정의하기
<details>
  <summary>1. 스키마 수정</summary>

#### (1) Tag 모델 추가 + 양쪽에 관계 필드
**prisma/schema.prisma**
```prisma
model User {
  id        Int       @id  @default(autoincrement())
  name      String
  todos     Todo[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  @@map("users")
}

model Todo {
  id         Int       @id  @default(autoincrement())
  title      String
  content    String?
  isDone     Boolean   @default(false)
  userId     Int
  user       User      @relation(fields: [userId], references: [id])
  tags       Tag[]     // 새로 추가: "이 Todo는 여러 Tag를 가진다"
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt

  @@map("todos")
}

model Tag {
  id    Int      @id  @default(autoincrement())
  name  String   @unique
  todos Todo[]   // "이 Tag는 여러 Todo에 붙는다"

  @@map("tags")
}
```

#### (2) N:M에서 주목할 점
```
N:M은 1:N과 다르게 양쪽 모델에 **배열 타입 필드**만 있으면 됩니다.

Todo 내 :  tags  Tag[]
Tag 내  :  todos Todo[]

둘 다 가상 필드 (실제 테이블에는 안 생김).
→ Prisma가 알아서 **중간 테이블 _TodoToTag**를 만들어주어요.

_TodoToTag 테이블 컴럼:
  A: Todo 의 id
  B: Tag  의 id

  → "Todo 1번이 Tag 5번을 가진다" = (A=1, B=5) 행 하나
```

#### (3) 마이그레이션 및 결과 확인
```
npx prisma migrate dev --name add_tag_and_n_to_n
```

**생성된 SQL 일부**
```SQL
-- CreateTable Tag
CREATE TABLE "tags" (
  "id"    SERIAL  NOT NULL,
  "name"  TEXT  NOT NULL,
  CONSTRAINT  "tags_pkey"  PRIMARY KEY ("id")
);
CREATE UNIQUE INDEX "tags_name_key" ON "tags"("name");

-- CreateTable 중간 테이블 (Prisma가 자동 생성!)
CREATE TABLE "_TodoToTag" (
  "A"  INTEGER NOT NULL,
  "B"  INTEGER NOT NULL
);

ALTER TABLE "_TodoTag"
  ADD CONSTRAINT "_TodoToTag_A_fkey" FOREIGN KEY ("A") REFERENCES "todos"("id") ON DELETE CASCADE,
  ADD CONSTRAINT "_TodoToTag_B_fkey" FOREIGN KEY ("B") REFERENCES "tags"("id") ON DELETE CASCADE; 
```
</details>

<details>
  <summary>2. 시드 확장</summary>

#### (1) Tag 생성 + Todo와 연결
**prisma/seed.js**
```JS
async function main() {
  await prisma.todo.deleteMany();
  await prisma.user.deleteMany();
  await prisma.tag.deleteMany();

  // 1) 사용자
  const alice = await prisma.user.create({ data: { name: 'Alice' } });
  const bob   = await prisma.user.create({ data: { name: 'Bob' } });

  // 2) 태그 먼저 생성
  const tagHome    = await prisma.tag.create({ data: { name: '집안일' } });
  const tagStudy   = await prisma.tag.create({ data: { name: '공부' } });
  const tagHealth  = await prisma.tag.create({ data: { name: '건강' } });

  // 3) Todo 생성 + tags 연결 (connect 문법)
  await prisma.todo.create({
    data: {
      title: '우유 사오기',
      userId: alice.id,
      tags: { connect: [{ id: tagHome.id }] }
    }
  });

  await prisma.todo.create({
    data: {
      title: 'Prisma 공부하기',
      userId: alice.id,
      tags: { connect: [{ id: tagStudy.id }] }
    }
  });

  await prisma.todo.create({
    data: {
      title: '운동하기',
      isDone: true,
      userId: alice.id,
      tags: { connect: [{ id: tagHealth.id }, { id: tagHome.id }] } // 태그 2개 동시에
    }
  });

  await prisma.todo.create({
    data: { title: '이메일 확인', isDone: true, userId: bob.id } // 태그 없이도 생성 가능
  });

  console.log('시드 완료');
}
```

# 3. User-Profile 일대일(1:1) 관계 정의하기
<details>
  <summary>1. 스키마 정의</summary>

#### (1) 1:1 스키마 작성
**prisma/schema.prisma**
```prisma
model User {
  id         Int       @id  @default(autoincrement())
  name       String
  profile    Profile?  // 1:1, 선택적 (Profile 없는 Usereh rksmd)
  todos      Todo[]
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt

  @@map("users")
}

model Profile {
  id        Int      @id  @default(autoincrement())
  bio       String?
  avatarUrl String?
  userId    Int      @unique  // 1:1의 핵심: @unique가 붙음
  user      User     @relation(fields: [userId], references: [id])

  @@map("profiles")
}
```

#### (2) 1:N과 1:1의 차이
```
[1:N] Todo                      [1:1] Profile
  userId  Int                     userId  Int  @unique  ← @unique 만 추가!

@unique 가 붙으면:
  → "같은 userId 가진 행이 두 개 존재할 수 없음"
  → DB가 컴입 이게 1:1 관계임을 알게 됨
  → 그래서 User 쪽도 Profile[] 이 아니라 Profile? 으로 적음

"한 사용자는 프로필 하나만" = userId 가 unique 해야 함.
```

#### (3) 마이그레이션
```
npx prisma migrate dev --name add_profile
```

**생성되는 SQL**
```SQL
CREATE TABLE "profiles" (
  "id"        SERIAL  NOT NULL,
  "bio"       TEXT,
  "avatarUrl" TEXT,
  "userId"    INTEGER  NOT NULL,

  CONSTRAINT  "profiles_pkey" PRIMARY KEY ("id")
);

-- 핵심: UNIQUE 제약이 외래키에 걸림
CREATE UNIQUE INDEX "profiles_userId_key" ON "profiles"("userId");

ALTER TABLE "profiles"
  ADD CONSTRAINT "profiles_userId_fkey"
  FOREIGN KEY ("userId") REFERENCES "users"("id")
  ON DELETE RESTRICT ON UPDATE CASCADE;
```
</details>

# 4. Relation의 onDelete 설정하기
<details>
  <summary>1. onDelete 옵션 종류</summary>

#### (1) 4가지 옵션
```
[Restrict / 기본값]   자식이 있으면 부모 삭제 불가 (에러 발생)
                       → User 가 Todo 를 가지고 있으면 User 삭제 못함

[Cascade]              부모 삭제 시 자식도 같이 삭제
                       → User 삭제하면 그 User의 Todo도 모두 삭제

[SetNull]              부모 삭제 시 자식의 외래키를 null 로
                       → User 삭제하면 Todo.userId 가 null 이 됨 (Todo는 남음)
                       → 단, userId 가 nullable (Int?) 이어야 함

[NoAction]             DB 에 위임 (PostgreSQL 은 Restrict 와 비슷)
```
</details>

<details>
  <summary>2. 스키마에 onDelete 추가</summary>

# 5. 관련된 객체 조회하기
<details>
  <summary>1. include - 관계 통채로 가져오기</summary>

#### (1) 기본 사용
```JS
// Todo와 함께 관련된 user, tags 조회
const todos = await prisma.todo.findMany({
  include: {
    user: true,
    tags: true
  }
});
```

#### (2) 중첩 조회(관계의 관계)
```JS
// Todo의 user의 다른 todos까지(3단 중첩)
const todos = await prisma.todo.findMany({
  include: {
    user: {
      include: { todos: true }  // 이 사용자의 다른 모든 Todo도 함께
    },
    tags: true
  }
});
```

</details>

<details>
  <summary>2. select - 원하는 필드만 콕 집기</summary>

#### (1) 기본 사용
```JS
// title과 isDone만 필요한 경우
const todos = await prisma.todo.findMany({
  select: {
    id: true,
    title: true,
    isDone: true
  }
});

// 결과:
// [{ id: 1, title: '우유 사오기', isDone: false }, ...]
// content, userId, createdAt 등은 아예 안 나옴
```

#### (2) select와 관계 같이 쓰기
```JS
const todos = await prisma.todo.findMany({
  select: {
    id: true,
    title: true,
    user: {
      select: { name: true } // user는 name만
    },
    tags: {
      select: { name: true } // tag도 name만
    }
  }
});

// 결과:
// [
//    { id: 1, title: '우유 사오기', user: { name: 'Alice' }, tags: [{ name: '집안일' }] },
//    ...
// ]

```

#### (3) include vs select 정리
```
include : "모든 기본 필드 + 관계도 추가로 가져와"
          (기본적으로 모든 컬럼 포함)

select  : "이 필드만 가져와"
          (지정한 컬럼만, 나머지는 안 줄)

⚠️ 같이 쓰면 에러:
  prisma.todo.findMany({ include: {...}, select: {...} })  ← ❌
  이유: 둘은 "전체 + 추가" vs "필요한 것만" 으로 사상이 다름
```
* 관리자 페이지처럼 많은 정보 다 보여주어야 할 때 : **`include`**
* 모바일/API 효율(필요한 것만 전송하고 싶을 때) : **`select`**
* 실무에서는 `select`가 더 권장됨(전송 데이터 최소화)

</details>

<details>
  <summary>3. 관계 필터링</summary>

#### (1) 관계 안에서 조건 처리(조건부 include)
```JS
// 사용자별로 "미완료" Todo 만 함께 가져오기
const users = await prisma.user.findMany({
  include: {
    todos: {
      where: { isDone: false }   // 관계 조회에 조건 추가
    }
  }
});

// 결과:
// [
//   { id: 1, name: 'Alice', todos: [ 그중 isDone=false 인 것만 ] },
//   { id: 2, name: 'Bob',   todos: [ ... ] }
// ]
```

#### (2) 관계 조건으로 부모 필터링(중요)
```JS
// '공부' 태그가 붙은 Todo만
const todos = await prisma.todo.findMany({
  where: {
    tags: {
      some: { name: '공부' }  // 태그 중 하나라도 "공부"인 Todo
    }
  }
});

// some  : 하나라도 일치(적어도 하나 이상)
// every : 모두 일치
// none  : 하나도 없음
```
</details>

* **`include`**: 전체 + 관계
* **`select`**: 원하는 것만
* **관계 안 `where`**: 필터 중첩

# 8. 관련된 객체 연결, 연결 해제하기
<details>

#### (1) 비교
```

[connect]    기존 연결 유지 + 새로 추가
             예: connect { id: 5 (오늘) }
             결과: [공부, 중요, 오늘]

[disconnect] 특정 관계만 끊기 (Tag 자체는 안 사라짐)

[set]        전부 끊고, 명시한 것만 연결

             결과: [급함]
             → 기존 [공부, 중요] 다 끊고 새로 [급함] 만

[deleteMany] 연결 끊으면서 Tag 자체도 삭제 (위험!)
```

# 10. 비즈니스 로직: Todo + Tag 함께 복사하기
<details>
  <summary>1. 요구사항</summary>

#### (1) 시나리오
```
[시나리오]
사용자가 "이 Todo 복사" 버튼을 눌렀다.
→ 원본 Todo 의 title, content, userId 를 이어 새 Todo 생성
→ 원본에 붙어 있던 Tag 들도 새 Todo 에 똑같이 연결
→ isDone 은 false 로 초기화 (복사한건 해야 할 일이니까)

API: POST /todos/:id/copy
```

#### (2) 순진한 구현 - 어디서 문제가 생기는건가?
```JavaScript
export async function copyTodo(sourceId) {
  // 1. 원본 조회 (태그 포함)
  const source = await prisma.todo.findUnique({
    where: { id: parseInt(sourceId) },
    include: { tags: true }
  });

  if (!source) throw new Error('Todo 없음');

  // 2. 새 Todo 생성 (태그 연결도 한 번에)
  const copied = await prisma.todo.create({
    data: {
      title: source.title + ' (복사본)',
      content: source.content,
      userId: source.userId,
      isDone: false,
      tags: {
        connect: source.tags.map(tag => ({ id: tag.id }))
      }
    },
    include: { tags: true }
  });

  return copied;
}
```
</details>

<details>
  <summary>2. 왜 트랜잭션이 필요한가</summary>

#### (1) "읽고-쓰기" 사이의 타이머
```
[문제 상황]
시각 T1: 원본 Todo 조회 (tag 5개 포함)
         findUnique({ include: { tags: true } })

         ↓ (이 사이에 다른 요청이 Tag 1개 삭제 했다면?)

시각 T2: 새 Todo 생성시 connect: [5개 id]
         ❌ 이미 삭제된 id 다이 있으면 에러!

→ T1 시점의 읽기가 T2 시점에 손상됨.

[해결]
T1 과 T2 를 "동일한 시점에서 일어난" 한 묶음으로 만들자. → 트랜잭션!
```
```shell
시간 →
T1 (copyTodo 트랜잭션 시작)
  ├─ findUnique(todo, include: tags)  → tags = [{id:1}, {id:2}, {id:3}]
  │
  │   ← 이 사이에 T2가 끼어듦
  │   T2: DELETE FROM tag WHERE id = 2  → COMMIT
  │
  ├─ todo.create({ tags: { connect: [{id:1},{id:2},{id:3}] } })
  │   → tag id=2 는 이제 없음!
  │   → 💥 외래키 제약 위반 에러 (Foreign Key Violation)
  └─ ROLLBACK (트랜잭션 전체 취소)
```

#### (2) ACID - 트랜잭션의 4가지 속성
```
A - Atomicity (원자성)   : 모두 성공 또는 모두 실패 (중간 없음)
C - Consistency (일관성) : DB 는 항상 유효한 상태만 갈 수 있음
I - Isolation (고립성)   : 동시에 수행되는 트랜잭션이 서로 간섭 안 함
D - Durability (지속성)  : 성공한 트랜잭션은 영구적으로 저장

우리가 주로 활용하는 건 A 와 C. 특히 A!
```
</details>
